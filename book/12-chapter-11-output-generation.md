# Chapter 11: Output Generation

A parser extracts structure from text. A semantic analyser validates that structure and resolves references. But neither produces anything a user can *use*. The final stage of a DSL pipeline is **output generation** — transforming the validated semantic model into a target format that downstream tools can consume. This chapter examines the principles and techniques of output generation, using MSD's production of `.merisio` JSON files as our running case study.

---

## 11.1 Transforming the Intermediate Representation

After parsing and semantic analysis, the pipeline holds a validated intermediate representation — a structured, in-memory model of everything the user declared. The output generator's job is to convert this model into the target format. Different DSLs have different targets:

- **SQL DSLs** generate DDL statements (`CREATE TABLE`, `ALTER TABLE`, `ADD CONSTRAINT`).

- **SASS/LESS** generate CSS stylesheets.
- **Terraform HCL** generates execution plans and API calls to cloud providers.
- **MSD** generates `.merisio` JSON files that the Merisio application can load.

Despite these differences, the underlying principle is the same: **the output generator should be a thin layer over the semantic model**. It should not perform validation, resolve references, or make design decisions — all of that belongs in earlier pipeline stages. The output generator's sole responsibility is serialisation: walking the model and writing it in the target format.

This separation of concerns is not merely aesthetic. It has practical consequences:

- **Testability**: the model can be inspected and tested independently of its serialised form.

- **Multiple outputs**: the same model can be serialised to JSON, SQL, SVG, or any other format by swapping the output generator.

- **Stability**: changes to the output format do not require changes to the parser or semantic analyser.

In MSD, the builder (Chapter 10) validates the parse result and produces a standard `Project` object. The output generator then calls `FileIO.save_project()` to write JSON. The builder never knows or cares what format the output will take. The serialiser never knows or cares how the model was constructed.

## 11.2 UUID Generation for Persistent Identity

Every element in a Merisio project — every entity, every association, every link — needs a unique identifier. These identifiers serve as stable references: a link connects an entity to an association by referencing their IDs, not their names. Names can be renamed; IDs persist.

### Why UUIDs?

MSD uses UUIDs (Universally Unique Identifiers) rather than sequential integers. The choice is deliberate:

- **No central counter needed.** Sequential IDs require a shared counter — who assigns ID 1? ID 2? With UUIDs, every object generates its own identifier independently.

- **Merge-friendly.** If two users create models independently and merge them, sequential IDs would collide. UUIDs are statistically unique across all models, on all machines, for all time.

- **Stable across re-imports.** When a `.merisio` file is saved and reloaded, the IDs are preserved exactly. The `from_dict()` class methods on `Entity`, `Association`, and `Link` all restore the original UUID from the JSON data.

### How UUIDs Appear

In MSD's implementation, UUIDs are generated by Python's `uuid` module via dataclass default factories. Here is the pattern, drawn directly from the `Entity` class:

```python
from dataclasses import dataclass, field
import uuid

@dataclass
class Entity:
    name: str
    attributes: list = field(default_factory=list)
    x: float = 0.0
    y: float = 0.0
    id: str = field(default_factory=lambda: str(uuid.uuid4()))
```

The `field(default_factory=lambda: str(uuid.uuid4()))` declaration means that every time an `Entity` is created *without* an explicit `id` argument, a fresh UUID is generated automatically. The builder never calls `uuid.uuid4()` directly — it simply writes `Entity(name=pe.name)` and the dataclass machinery handles the rest. The same pattern applies to `Association` and `Link`.

### The Determinism Concern

UUIDs are random. Re-parsing the same MSD source file produces different UUIDs each time, because new `Entity` and `Association` objects are created with fresh random IDs. This means two `.merisio` files generated from the same `.msd` source will not be byte-identical.

This is acceptable for two reasons:

1. **The `.merisio` file preserves IDs once saved.** After the first save, the JSON contains explicit IDs. Reloading and re-saving preserves them.

2. **The semantic content is identical.** The entity names, attributes, links, and cardinalities are the same — only the internal identifiers differ. For version control purposes, the `.msd` source file is the canonical artefact, not the generated `.merisio` output.

> **Note:** If deterministic output were required — for example, if the generated file needed to be committed to version control — the UUIDs could be derived from a hash of the entity name plus a project-level seed. MSD deliberately avoids this complexity because the `.msd` source, not the `.merisio` output, is the version-controlled artefact.

## 11.3 Metadata Application and Defaults

The `project {}` block in an MSD file provides optional metadata: a project name, an author, and a description. The builder applies this metadata to the `Project` object during construction:

```python
# Apply metadata
if parse_result.metadata:
    m = parse_result.metadata
    if m.name:
        project.name = m.name
    if m.author:
        project.author = m.author
    if m.description:
        project.description = m.description
```

Several details are worth noting:

- **Metadata is optional.** If the MSD file contains no `project {}` block, `parse_result.metadata` is `None`, and the `Project` retains its defaults (`"Untitled Project"` for the name, empty strings for author and description).

- **Empty values are not applied.** If the user writes `name:` with no value, the parsed metadata will have `name = ""`. The `if m.name:` guard prevents this empty string from overwriting the default. This preserves the principle of least surprise: omitting a field and leaving it blank have the same effect.

- **The builder owns the application logic.** The output generator does not need to understand metadata semantics — it simply serialises whatever the `Project` object contains.

This pattern — *if present and non-empty, apply; otherwise, preserve defaults* — is common across DSL output pipelines. It keeps the metadata system additive: users can provide as much or as little metadata as they wish, and the output is always valid.

## 11.4 JSON Serialisation for Tool Interoperability

MSD's target output is the `.merisio` file format — a JSON document that the Merisio application can load directly. The choice of JSON as the serialisation format was not arbitrary.

### Why JSON?

- **Universal.** Every programming language has a JSON parser. The `.merisio` file can be read by Python, Go, JavaScript, or any other language.

- **Human-readable.** A developer can open a `.merisio` file in a text editor and inspect its contents. This is invaluable for debugging.

- **Version-controllable.** JSON diffs are meaningful in Git. Adding an entity or changing a cardinality produces a clean, reviewable diff.

- **Schema-flexible.** New fields can be added without breaking older readers (they simply ignore unknown keys). This supports forward compatibility as Merisio evolves.

### The .merisio Structure

The `.merisio` format stores the complete project state. The `Project.to_dict()` method produces the following structure:

```python
def to_dict(self) -> dict:
    self.modified_at = datetime.now().isoformat()
    return {
        "version": self.VERSION,
        "metadata": {
            "name": self.name,
            "description": self.description,
            "author": self.author,
            "created_at": self.created_at,
            "modified_at": self.modified_at
        },
        "mcd": {
            "entities": [e.to_dict() for e in self._entities.values()],
            "associations": [a.to_dict() for a in self._associations.values()],
            "links": [l.to_dict() for l in self._links.values()]
        },
        "mld": {
            "column_overrides": self._mld_column_overrides
        },
        "colors": self.colors
    }
```

And `FileIO.save_project()` writes it as formatted JSON:

```python
@staticmethod
def save_project(project: Project, file_path: str) -> bool:
    try:
        data = project.to_dict()
        with open(file_path, 'w', encoding='utf-8') as f:
            json.dump(data, f, indent=2, ensure_ascii=False)
        project.file_path = file_path
        project.modified = False
        return True
    except Exception as e:
        print(f"Error saving project: {e}")
        return False
```

### Reusing Existing Serialisation

A key benefit of building a standard `Project` object is that the MSD pipeline reuses Merisio's existing serialisation code unchanged. The builder produces a `Project`; `FileIO.save_project()` writes it to disk. No special MSD-specific serialiser is needed. This is a significant advantage: every downstream tool that works with `.merisio` files — the GUI, the CLI, the SQL generator, the diagram exporter — works automatically with MSD-generated projects.

> **Tip:** When designing an output generator for your DSL, aim to produce the same model object that the rest of your toolchain already uses. If your GUI application has a `Project` class with a `save()` method, make your DSL builder produce a `Project`. You get serialisation, loading, validation, and all downstream transformations for free.

## 11.5 Round-Trip Fidelity

A critical quality metric for any DSL output pipeline is **round-trip fidelity**: the ability to parse a source file, generate the output, reload that output, and verify that no information was lost.

### The Round-Trip Test

For MSD, the round-trip is:

1. **Parse** an `.msd` file into a `ParseResult`.
2. **Build** a `Project` from the `ParseResult`.
3. **Save** the `Project` as a `.merisio` JSON file.
4. **Load** the `.merisio` file back into a new `Project`.
5. **Verify** that the loaded project matches the original.

### What Is Preserved

The round-trip preserves all *semantic* content:

- **Entity names** and their attributes (names, data types, sizes, primary key status).

- **Association names** and their carrying attributes.

- **Link connections** (which entity connects to which association) and their cardinalities.

- **Metadata** (project name, author, description).
- **Positions** (x, y coordinates from auto-layout).

### What Is NOT Preserved

Certain source-level concerns are deliberately lost:

- **Comments.** MSD supports `#` and `//` comments, but these are discarded by the lexer and never reach the model. The `.merisio` format has no comment storage.

- **Whitespace and formatting.** The original indentation, blank lines, and spacing are not part of the semantic model.

- **Original ordering.** Entities may appear in a different order in the JSON than they did in the MSD source, because the `Project` stores them in a dictionary keyed by UUID.

- **Source location information.** Line numbers and column offsets exist only during parsing and building — they are not serialised.

These losses are acceptable because the `.msd` source file remains available. The `.merisio` output is a *derived artefact*, not the source of truth. If the user needs comments or specific ordering, they maintain the `.msd` file and regenerate the `.merisio` as needed.

### Testing Round-Trip Fidelity

A round-trip test follows a straightforward pattern:

```python
def test_round_trip(msd_source: str, tmp_path):
    # Parse and build
    parser = MSDParser()
    result = parser.parse(msd_source)
    builder = MSDProjectBuilder()
    project, errors = builder.build(result)
    assert not any(e.severity == "error" for e in errors)

    # Save
    out_path = str(tmp_path / "test.merisio")
    FileIO.save_project(project, out_path)

    # Reload
    loaded = FileIO.load_project(out_path)
    assert loaded is not None

    # Verify entities
    orig_entities = project.get_all_entities()
    loaded_entities = loaded.get_all_entities()
    assert len(orig_entities) == len(loaded_entities)

    for orig in orig_entities:
        match = loaded.get_entity_by_name(orig.name)
        assert match is not None
        assert len(match.attributes) == len(orig.attributes)
        for oa, la in zip(orig.attributes, match.attributes):
            assert oa.name == la.name
            assert oa.data_type == la.data_type
            assert oa.size == la.size
            assert oa.is_primary_key == la.is_primary_key
```

If any assertion fails, the pipeline has a bug — either something was lost during serialisation or corrupted during deserialisation. Round-trip tests are among the most valuable tests in a DSL implementation because they exercise the entire pipeline end to end.

## 11.6 Output Formats for Other DSLs

While MSD generates JSON, other DSLs target radically different output formats. The general pattern remains the same — *DSL source to parsed model to validated model to target format* — but the serialisation concerns vary.

### SQL DSLs

A DSL that describes database schemas (MSD itself is one) might generate PostgreSQL DDL:

```sql
CREATE TABLE student (
    student_id INT,
    name VARCHAR(100),
    email VARCHAR(255),
    PRIMARY KEY (student_id)
);
```

The output generator walks each entity, maps attribute types to SQL types, identifies primary keys, and emits `CREATE TABLE` statements. Foreign keys are derived from link cardinalities. Merisio's `SQLGenerator` class demonstrates this pattern — it takes a `Project` and produces a complete DDL script.

### Configuration DSLs

A DSL for application configuration might output YAML, TOML, or environment variables:

```yaml
database:
  host: localhost
  port: 5432
  name: university
```

Here, the output generator must handle nesting, quoting rules, and format-specific escaping. The semantic model is flat (key-value pairs with optional grouping), but the output format has its own syntax.

### Diagram DSLs

A DSL for diagrams might output SVG, PNG, or PDF. Merisio's `HeadlessRenderer` demonstrates this: it creates a `QGraphicsScene`, adds `EntityItem` and `AssociationItem` widgets using the same paint logic as the GUI, and exports the scene to the target format. The semantic model is the same `Project` object — only the output generator differs.

### Query DSLs

A DSL for queries might output API calls, SQL subqueries, or execution plans. The output is not a file but a runtime action. The "serialisation" is the translation from the declarative query model to the imperative operations the runtime must perform.

### The Common Pattern

Regardless of the target format, the pattern holds:

```
DSL source → Lex → Parse → Validate → Build Model → Serialise to Target
```

Each stage has a single responsibility. The serialiser knows the target format's syntax and conventions. The model knows the domain semantics. Neither intrudes on the other's territory.

## 11.7 MSD: Building a Merisio Project from ParseResult

Let us walk through the complete MSD output pipeline, from parsed tokens to saved JSON.

### Step 1: Create the Project Object

The builder begins with a fresh `Project`:

```python
project = Project()
```

This creates an empty project with default metadata (`"Untitled Project"`), empty entity/association/link dictionaries, default diagram colours, and timestamps set to the current time.

### Step 2: Apply Metadata

If the parse result contains a `project {}` block, its values are applied:

```python
if parse_result.metadata:
    m = parse_result.metadata
    if m.name:
        project.name = m.name
    if m.author:
        project.author = m.author
    if m.description:
        project.description = m.description
```

### Step 3: Create Entities with Attributes

For each parsed entity, the builder creates an `Entity` object (which automatically receives a UUID) and populates its attributes:

```python
for pe in parse_result.entities:
    entity = Entity(name=pe.name)
    for pa in pe.attributes:
        attr = Attribute(
            name=pa.name,
            data_type=pa.data_type,
            size=pa.size,
            is_primary_key=pa.is_primary_key,
        )
        entity.add_attribute(attr)
    project.add_entity(entity)
```

### Step 4: Create Associations

Associations follow the same pattern. They too receive automatic UUIDs and may carry their own attributes:

```python
for pa in parse_result.associations:
    assoc = Association(name=pa.name)
    for attr_parsed in pa.attributes:
        attr = Attribute(
            name=attr_parsed.name,
            data_type=attr_parsed.data_type,
            size=attr_parsed.size,
            is_primary_key=attr_parsed.is_primary_key,
        )
        assoc.add_attribute(attr)
    project.add_association(assoc)
```

### Step 5: Create Links with Resolved References

Links are where name resolution meets output generation. The builder has already constructed dictionaries mapping entity names to `Entity` objects and association names to `Association` objects. Each link resolves its text-based references to UUID-based references:

```python
entity = entity_names.get(pl.entity_name)
assoc = assoc_names.get(pl.association_name)

link = Link(
    entity_id=entity.id,
    association_id=assoc.id,
    cardinality_min=pl.cardinality_min,
    cardinality_max=pl.cardinality_max,
)
project.add_link(link)
```

The link stores `entity.id` and `assoc.id` — the UUIDs — not the names. This is the critical transformation: the output format uses identity-based references, while the source format uses name-based references. The builder bridges the two.

### Step 6: Auto-Layout

The builder invokes the Fruchterman-Reingold layout algorithm (Chapter 12) to assign x, y positions to all entities and associations. This ensures that the diagram will render without overlapping nodes when opened in the Merisio GUI.

### Step 7: Mark as Clean and Save

Finally, the builder marks the project as unmodified and returns it:

```python
project.modified = False
return project, errors
```

The caller — whether the CLI tool or the GUI's import function — then saves the project:

```python
FileIO.save_project(project, "university.merisio")
```

> **Warning:** When designing your output format, always include a version field (here, `"version": "2.1"`). Without it, future changes to the format become breaking changes with no way to detect or migrate older files. The version field is cheap to add and invaluable when you need to evolve the format.
